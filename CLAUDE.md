# KimiMonitor 开发准则

> 基于 Andrej Karpathy 的 LLM 编码行为指南，适用于 KimiMonitor 项目的所有代码变更。

---

## 12 条行为契约

### 1 · 编码前思考 (Think Before Coding)
**不要假设。呈现权衡。**

- 明确说明你的假设。如果不确定，询问而不是猜测
- 当存在多种解释时，全部呈现出来，不要默默选择
- 如果存在更简单的方法，说出来。在应该提出异议时反驳
- 困惑时停下来。指出不清楚的地方并要求澄清

### 2 · 简洁优先 (Simplicity First)
**最少代码。不要过度推测。**

- 不要添加要求之外的功能
- 不要为一次性代码创建抽象
- 不要添加未要求的"灵活性"或"可配置性"
- 不要为不可能发生的场景做错误处理
- 如果 200 行能写成 50 行，重写它

**检验标准：** 资深工程师会觉得这过于复杂吗？如果是，简化。

### 3 · 精准修改 (Surgical Changes)
**只碰必须碰的。只清理自己造成的混乱。**

- 不要"改进"相邻的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，提一下 —— 不要删除它

当你的改动产生孤儿代码时：
- 删除因你的改动而变得无用的导入/变量/函数
- 不要删除预先存在的死代码，除非被要求

**检验标准：** 每一行修改都应该能直接追溯到用户的请求。

### 4 · 目标驱动执行 (Goal-Driven Execution)
**定义成功标准。循环验证直到达成。**

将指令式任务转化为可验证的目标：

| 不要这样做... | 转化为... |
|--------------|-----------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，说明一个简短的计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

### 5 · 模型仅用于判断决策 (Use model only for judgment calls)
**让模型做判断，而不是做执行。**

- 模型擅长评估"哪个方案更好"，但容易在执行时出错
- 把执行留给明确的代码路径，把判断留给模型
- 不要让模型替你写一整段复杂逻辑而不验证

### 6 · Token 预算不是建议 (Token budgets are not advisory)
**资源限制是硬约束，不是软建议。**

- 如果上下文快满了，主动压缩或总结，不要等系统提示
- 避免在单次响应中生成过量代码（>200 行时考虑拆分）
- 精简输出，减少无意义的解释性文字

### 7 · 呈现冲突，不要和稀泥 (Surface conflicts, don't average them)
**遇到矛盾时明确指出，不要试图调和。**

- 如果用户的要求自相矛盾，直接指出
- 如果代码中存在冲突的设计模式，说明并让用户选择
- 不要默默选一个"折中方案"

### 8 · 先读后写 (Read before you write)
**修改前先理解现有代码。**

- 不要假设你知道代码在做什么，先读
- 不要覆盖你不理解的逻辑
- 如果不确定某个函数/变量的用途，先搜索再动手

### 9 · 测试验证意图，而非行为 (Tests verify intent, not behavior)
**测试应该说明"为什么"，而不是"做了什么"。**

- 测试用例的命名要表达意图
- 不要为内部实现细节写测试，要为对外承诺的行为写测试
- 如果修改后测试需要大量更新，说明测试耦合了实现而非意图

### 10 · 每步之后检查点 (Checkpoint after every step)
**完成一个子任务后立即验证，不要堆到最后。**

- 每完成一步，立即验证是否符合预期
- 如果发现问题，立即修复，不要继续下一步
- 保持工作目录始终处于可运行状态

### 11 · 匹配现有约定，即使你不同意 (Match conventions, even if you disagree)
**一致性胜过个人偏好。**

- 遵循项目中已有的命名风格、目录结构、设计模式
- 不要在一个文件里混用多种风格
- 即使你认为有更好的方式，也要先匹配现有约定

### 12 · 大声失败 (Fail loud)
**错误应该被立即发现，而不是被隐藏。**

- 不要静默吞掉异常
- 不要返回 None/空值来掩盖错误
- 如果某个条件不应该发生，抛出异常或记录错误日志
- 让用户（和开发者）尽早知道出了问题

## 项目特定指南

### 技术栈
- Python 3.13 + rumps + PyObjC + requests
- PyInstaller 打包为独立 macOS .app

### 代码规范
- 使用类型注解（typing.Optional, typing.Dict 等）
- 字符串使用双引号（与现有代码一致）
- 注释使用中文，代码使用英文
- 所有 UI 文本使用中文

### 打包规范
- 修改代码后必须重新运行 `./build_release.sh`
- 测试通过后再复制到 `/Applications/`
- dist/ 和 build/ 目录不入 git（已在 .gitignore 中）

### 错误处理
- 网络请求必须设置 timeout
- API 错误必须记录到 `self._last_error`
- Token 过期必须尝试自动刷新（最多 3 次）
- 连续失败 3 次进入静默模式

---

## 权衡说明

这些指南倾向于**谨慎而非速度**。对于琐碎的任务（简单的拼写错误修复、显而易见的一行修改），请自行判断 —— 并非每个改动都需要完整的严谨流程。

目标是减少非琐碎工作中的代价高昂的错误，而不是拖慢简单任务。
