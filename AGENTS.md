## 12 条行为契约

### 1 · 编码前思考
**不要假设。呈现权衡。**

- 明确说明假设。不确定时询问而非猜测
- 存在多种解释时全部呈现，不要默默选择
- 存在更简单方法时说出来，该反驳时反驳
- 困惑时停下来，指出不清楚的地方并要求澄清

### 2 · 简洁优先
**最少代码。不要过度推测。**

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未要求的"灵活性"
- 不为不可能场景做错误处理
- 200 行能写成 50 行就重写

### 3 · 精准修改
**只碰必须碰的。只清理自己造成的混乱。**

- 不"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格
- 你的改动产生的孤儿代码要删除
- 预先存在的死代码只提不删

### 4 · 目标驱动执行
**定义成功标准。循环验证直到达成。**

每步之后检查点：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
```

### 5 · 模型仅用于判断决策
让模型做判断，执行留给明确代码路径。

### 6 · Token 预算不是建议
上下文快满时主动压缩，单次响应避免 >200 行代码。

### 7 · 呈现冲突，不要和稀泥
要求自相矛盾时直接指出，不默默选折中方案。

### 8 · 先读后写
修改前先理解现有代码，不确定时先搜索再动手。

### 9 · 测试验证意图，而非行为
测试命名表达意图，不耦合实现细节。

### 10 · 每步之后检查点
完成子任务立即验证，保持工作目录始终可运行。

### 11 · 匹配现有约定，即使你不同意
一致性胜过个人偏好，不混用多种风格。

### 12 · 大声失败
不静默吞异常，不返回 None 掩盖错误，尽早暴露问题。

## 项目特定规范

### 技术栈
- Python 3.13 + rumps + PyObjC + requests
- PyInstaller 打包独立 macOS .app

### 代码风格
- 类型注解必须（typing.Optional, typing.Dict 等）
- 字符串用双引号
- 注释中文，代码英文
- UI 文本中文

### 开发流程
1. 修改代码 → 运行 `./build_release.sh` 打包
2. 验证 dist/KimiMonitor.app 正常 → 复制到 /Applications/
3. 提交 git（dist/ build/ 不入版本控制）

### 错误处理红线
- 网络请求必须设 timeout
- API 错误必须记录到 `self._last_error`
- Token 过期必须尝试自动刷新（最多 3 次，见 app.py 熔断机制）
- 连续失败 3 次进入静默模式
